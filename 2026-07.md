## concern

**`ActiveSupport::Concern`**（Concern）は、複数のクラスで共通利用する処理をモジュール化し、再利用しやすくする仕組みである。

Railsでは慣習として、

- app/controllers/concerns
- app/models/concerns

に配置することが多い。

Concernは通常のModuleを拡張したもので、

- included do
- class_methods do

を利用して、include先へクラスメソッドや設定を追加できる。

## 類似

### Helper

Viewで利用する補助メソッドをまとめる仕組み。

例)

- 日付のフォーマット
- 金額の表示形式

### Decorator

ModelのデータをView向けに整形・装飾するオブジェクト。

Viewに表示ロジックを書きすぎないために利用する。

例)

- ユーザーのフルネーム表示
- 投稿日時の表示形式

## 正規表現

### 正規表現リテラル

Rubyでは、`/.../` で正規表現（Regexp）を表す。

```ruby
/abc/

# ↓ と同じ
Regexp.new("abc")
```

`"abc"` が文字列リテラルであるのに対し、

```ruby
/abc/
```

は **正規表現リテラル** と呼ばれる。

#### 正規表現リテラルの構成

正規表現リテラルの中では、様々な構文を組み合わせて検索条件を表現する。

```
正規表現リテラル (/.../)
├── 文字をそのまま書く
│   └── /abc/
│
├── 文字クラス
│   └── /[abc]/
│
├── POSIX文字クラス
│   └── /[[:space:]]+/
│
├── 文字種メタ文字
│   └── /\s+/
│
├── 量指定子
│   └── + * ? {n,m}
│
├── グループ化
│   └── (abc)
│
└── アンカー
    └── ^ $
```

### 文字クラス

```ruby
/[abc]/
```

文字クラスは、指定した文字のどれか1文字にマッチする。

例

```ruby
/[abc]/
```

↓

```
a または b または c
```

### POSIX文字クラス

```ruby
/[[:space:]]+/
```

の `[:space:]` は **POSIX文字クラス** と呼ばれる。

`[[:space:]]` は空白文字（半角スペース、タブ、改行など）にマッチする。

#### `[[:space:]]` の構造

```
[
  [:space:]
]
```

- 外側の `[ ]`
    - 文字クラス
- 内側の `[:space:]`
    - POSIX文字クラス名

#### 主なPOSIX文字クラス

```ruby
[[:space:]]  空白文字
[[:digit:]]  数字
[[:alpha:]]  アルファベット
[[:alnum:]]  英数字
```

### 文字種メタ文字

```ruby
/\s+/
```

の `\s` は **文字種メタ文字** と呼ばれる。

#### 主な文字種メタ文字

```ruby
\s   空白文字
\d   数字
\w   英数字と _
```

## normalizes

### 概要

Active Recordの属性値の正規化（normalization）を宣言するメソッド。

`validates` メソッドのようにクラス定義の中で宣言的に使用できる。

例)

```ruby
class User < ApplicationRecord
  normalizes :email, with: ->(email) { email.downcase }
end
```

### 適用タイミング

- 属性への代入や更新時に適用され、**正規化済みの値が DB に保存される**
- **finder メソッドのキーワード引数**（`find_by` / `find_or_create_by` / `find_or_initialize_by` 等の検索条件）にも適用される → 重複防止のための「保存値と検索キーの一致」が自動で担保される

### 備考

- Rails 7.1で導入されたメソッドである。
- デフォルトでは `nil` 値はスキップされる(正規化が走らない)
- `normalizes` は ActiveRecord 限定の機能
    - `ActiveModel::Model` ベースのフォームオブジェクトでは使えない。
    - FormObject 側で同等のことをしたい場合は、`ActiveModel::Validations::Callbacks` を include して `before_validation` で自前正規化する。

## SimpleCovの設定

- `Gemfile`にSimpleCovを追加し、`bundle install`を実行。
- テストヘルパーの**最上部**（`spec/rails_helper.rb`等）にSimpleCovを読み込み、起動する。
    
    ```ruby
    require 'simplecov'
    SimpleCov.start
    ```
    
    - SimpleCovはRails読み込み前に起動する必要がある。
        - Rails環境の読み込み時には、モデルクラス定義やinitializerなど、一度だけ実行されるコードが存在する。SimpleCov起動前にRailsを読み込むと、Railsロード時に実行されたコードが正しくカバレッジ計測されない。
        - 例) `sake.rb`
            
            ```ruby
            class Sake < ApplicationRecord
              PRODUCT_NAME_MAX_LENGTH = 255           # ← require 時に実行される
              normalizes_text :product_name           # ← require 時に実行される
              validates :product_name, presence: true # ← require 時に実行される
              scope :search_by_product_name, ...      # ← require 時に実行される
            end
            ```
            
            このコードは `Sake` クラスがrequireされた時点でRubyにより実行される。そのためSimpleCovはRailsロードより前に開始する。
            

### 補足

#### spec/rails_helper.rbの読み込みの流れ

```ruby
require "simplecov"                          # ① Rails はまだロードされていない
SimpleCov.start "rails" do ... end           #    （だからここでカバレッジ計測を開始）

require "spec_helper"                         # ② RSpec の基本設定
ENV["RAILS_ENV"] ||= "test"                  #    環境を test に固定
require_relative "../config/environment"     # ★③ ここで Rails アプリ全体が読み込まれる
abort(...) if Rails.env.production?           #    ↑以降は Rails / app のクラスが使える
require "rspec/rails"                         # ④ Rails 連携の RSpec 機能を読み込む
# Add additional requires below this line.    
require "webmock/rspec"                       # ⑤ Rails ロード後に追加の require

```

- 冒頭でSimpleCovを読み込まないと、SimpleCovの計測開始前に実行されたコードはカバレッジとして正しく記録されない

### 参考

- https://github.com/simplecov-ruby/simplecov/tree/v0.22.0

## Shoulda Matchers概要

Shoulda Matchersとは、Railsのモデルや関連などに対するテストを、RSpecで簡潔に記述するためのマッチャーライブラリ。

通常RSpecで複数行必要な検証を、1行の期待値として書けるようになる。

### マッチャー例

- マッチャー一覧
    - https://github.com/thoughtbot/shoulda-matchers#matchers
- ActiveModelマッチャー
    - `validate_presence_of`
        - 対象属性にpresence validationが設定されていることを検証する。
        - nilや空文字など、存在しない値を無効として扱うことを確認する。
    - `validate_length_of`
        - length validationが設定されていることを検証する。
        - minimum / maximumなどの文字数制限を確認できる。
        - https://matchers.shoulda.io/docs/v8.0.1/Shoulda/Matchers/ActiveModel.html#validate_length_of-instance_method
- ActiveRecord マッチャー
    - `validate_uniqueness_of`
        - uniqueness validationが設定されていることを検証する。
        - 同じ値を持つレコードが存在する場合、validationエラーになることを確認する。

### 注意

Shoulda Matchersはvalidationの設定を確認するもので、DB制約(unique indexなど)を保証するものではない。

### 参照

- https://github.com/thoughtbot/shoulda-matchers
- https://matchers.shoulda.io/docs/v8.0.1/
- https://www.miracleave.co.jp/contents/1487/rspec-shoulda-matchers/#index_id1

## RSpceのchangeマッチャの使い方

### 使い方

- 基本的な使い方：`expect { 何かの処理 }.to change { 監視したい値 }`
- 誤り：`expect(何かの処理).to change { 監視したい値 }`
    - `expect(何かの処理)` の時点で処理が実行されてしまう。
    - `change` マッチャは、ブロック内の処理の実行前後で値を比較するため、`expect()` に処理を書いてしまうと変更前の値を取得できず、前後の比較ができない。

### 補足

`change` マッチャは、次のような流れで動作する。

1. 監視対象の現在の値を取得する
2. `expect { ... }` のブロック内の処理を実行する
3. 監視対象の値を再度取得する
4. 前後の差分を比較する

そのため、変更を検証したい処理は必ずブロック (`{ ... }`) に書く。

## RSpec_モック・スタブの作成

| 書き方 | 作成 | 振る舞いを差し替える | 呼ばれたことを検証する |
| --- | --- | --- | --- |
| `allow(obj).to receive(:foo)` | スタブ | ○ | × |
| `expect(obj).to receive(:foo)` | モック（Message Expectation） | ○ | ○ |

### 補足

- `allow`
    
    ```
    allow(user).toreceive(:name).and_return("Alice")
    ```
    
    - `name` が呼ばれたら `"Alice"` を返す
    - 呼ばれなくてもテストは成功
- `expect`
    
    ```
    expect(user).toreceive(:name).and_return("Alice")
    ```
    
    - `name` が呼ばれたら `"Alice"` を返す
    - **必ず `name` が呼ばれることを検証する**
    - 呼ばれなければテスト失敗

## `send` メソッド

`send` は **Ruby標準のメソッド**（`Object`クラスのメソッド）で、メソッド名を動的に指定して呼び出すために使用する。

### 基本構文

```ruby
object.send(:method_name, 引数...)
```

### 例

```ruby
method_name = :upcase

"ruby".send(method_name)
# => "RUBY"
```

### private メソッドの呼び出し

`send` は 可視性（public/private/protected）を無視してメソッドを呼び出すため、**private メソッドも呼び出せる**。

```ruby
client.send(:fetch_data, :areas)
```

RSpecでは、**private メソッドをテストする際によく使用される**。

### `public_send` との違い

| メソッド | private を呼べる？ | 用途 |
| --- | --- | --- |
| `send` | ✅ 呼べる | メソッド名を動的に指定して呼び出す（可視性を無視） |
| `public_send` | ❌ 呼べない | メソッド名を動的に指定して呼び出す（publicのみ） |

## `RuntimeError`

`RuntimeError` は **Ruby標準の例外クラス**で、実行時に発生する一般的なエラーを表す。

`raise` に例外クラスを指定しない場合は、デフォルトで `RuntimeError` が発生する。

```ruby
raise "エラー"
# ↓ と同じ
raise RuntimeError, "エラー"
```

### Rubyのエラー体系

```markdown
Exception
├── StandardError
│   ├── RuntimeError
│   ├── ArgumentError
│   ├── NoMethodError
│   ├── TypeError
│   ├── ZeroDivisionError
│   └── ...
├── ScriptError
├── SystemExit
├── NoMemoryError
└── ...
```

### Javaとの対応イメージ

| Java | Ruby |
| --- | --- |
| `Throwable` | `Exception` |
| `Exception` | `StandardError` に近い |
| `RuntimeException` | `RuntimeError` |

## `include` マッチャ

`include` は、**指定した要素が含まれていること**を確認するRSpecのマッチャ。

- Hash：指定したキー・値の組が含まれていることを確認
- Array：指定した要素が含まれていることを確認
- String：指定した文字列が含まれていることを確認

```ruby
expect(hash).to include("id" => 1, "name" => "岩手")
```

→ Hash全体ではなく、指定したキー・値が含まれていればテストは成功する。

### `eq` との違い

| マッチャ | 比較方法 |
| --- | --- |
| `include` | 指定した要素（キー・値など）が含まれていればOK |
| `eq` | オブジェクト全体が完全一致している必要がある |

```ruby
hash = {
  "id" => 1,
  "name" => "岩手",
  "code" => "03"
}

expect(hash).to include("id" => 1, "name" => "岩手") # 成功
expect(hash).to eq({ "id" => 1, "name" => "岩手" }) # 失敗
```

`include` は必要な要素だけを確認したい場合、`eq` はオブジェクト全体を比較したい場合に使用する。

## RSpec Hooks (フック)

`before` や `after` は、**テストの実行前後に共通処理を実行するためのRSpecのフック（ライフサイクルフック）**。

### 主なフック

| フック | 実行タイミング |
| --- | --- |
| `before`（=`before(:each)`） | 各 `it` の実行前 |
| `after`（=`after(:each)`） | 各 `it` の実行後 |
| `before(:context)`（=`before(:all)`） | `describe` / `context` ごとに1回、最初に実行 |
| `after(:context)`（=`after(:all)`） | `describe` / `context` ごとに1回、最後に実行 |

### 例

```ruby
before do
  create(:area)
end
```

→ 各 `it` の実行前に `create(:area)` が実行される。

### JUnit4との対応

| JUnit4 | RSpec |
| --- | --- |
| `@Before` | `before` |
| `@After` | `after` |
| `@BeforeClass` | `before(:context)` |
| `@AfterClass` | `after(:context)` |

> Rails + RSpecでは、`before`（=`before(:each)`）を使用することが多い。

## `reload`

`reload` は **Active Record のメソッド**で、**データベースから最新の状態を再読み込みする**。

主に、他の処理でレコードが更新された後に、最新の値を取得したい場合に使用する。

```ruby
user.reload
```

`reload` はオブジェクト自身を返すため、メソッドチェーンで書ける。

```ruby
expect(user.reload.name).to eq("Alice")
```

> `update_all` などでDBを直接更新した場合、メモリ上のオブジェクトは更新されないため、`reload` が必要になることがある。
>
